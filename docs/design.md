# VPS監視デスクトップガジェット 設計書

- **アプリ名**: **VPS監視-Mei**（以下、本書で `VpsWatcher` とあるのは内部識別子・パス名）
- **作成日**: 2026-06-02（rev.6：アプリ名確定・開発体制/GitHub運用 §15 追加）
- **対象環境**: Windows 11（デスクトップ常駐） / Ubuntu VPS ×N（Apache/Nginx・PHP運用中、**Nは可変・今後増加し得る**）
- **クライアント技術**: C# / WPF（.NET 最新LTS）
- **サーバ側エージェント**: 言語非依存。**リファレンス実装＝Go静的バイナリ**、現環境向け代替＝PHP CLI
- **配布方針**: GitHub Releasesから直接ダウンロードして即実行（CIで自動ビルド）
- **音声**: VOICEPEAK（事前レンダリングWAVを同梱。クレジット表記不要のため採用）

---

## 0. 基本設計原則

1. **低負荷を最優先（サーバ・ローカルPC両方）** — このガジェットは24時間常駐する“監視される側に負担をかけない監視”であるべき。サーバ側は監視対象VPSの本来の仕事（ViPass・EC等）を圧迫してはならず、ローカル側も他作業の邪魔をしない。設計の各所で「アイドル時CPUほぼ0%・メモリ最小」を判断基準にする（具体策は §13 に集約）。
2. **公開ポートを増やさない** — 既存SSH（49222・鍵認証・fail2ban）のみを使う。
3. **アップデート耐性** — サーバ＝静的バイナリ、クライアント＝self-contained単一ファイルで、ランタイム依存を持たない。
4. **配布容易性** — GitHubから落として即動く。インタプリタ／ランタイムの追加インストールを要求しない。
5. **台数可変（N台スケール）** — 監視対象VPSは設定追加だけで増やせる。ハードコードされた台数前提を持たず、パネルはリストとして動的生成・並べ替え可能にする。
6. **YAGNI** — Phase 1は最小機能。拡張は布石だけ残す。
7. **既知の事実と推定の明示的分離** — 不確実な箇所は推定と明記する。

## 1. 目的とスコープ

N台のVPS（CPU・メモリ・ストレージ・ネットワーク）の状況を、Windowsデスクトップ上に常駐するガジェットでほぼリアルタイム（1Hz）に表示する。状態はAI生成イラストの表情差分で示し（CrystalDiskInfoの水晶雫方式）、しきい値を跨いだときにキャラボイスで警告する。監視対象は設定追加だけで増やせ、表示パネルはマウスのドラッグで並べ替えられる。各メトリクスはクライアント側で履歴を保持し、クリックでポップアップグラフを表示できる。

**Phase 1（MVP）でやること**
- VPS **N台**（初期は2台想定、設定追加で拡張可）のメトリクス収集と1秒間隔の取得
- 常駐ウィンドウに数値＋バー表示。**最前面常駐のON/OFFを設定で選択可**
- サーバパネルを**マウスドラッグで並べ替え**（gong-wpf-dragdrop採用、順序は永続化）
- **メトリクス履歴のクライアント側保持＋クリックでポップアップグラフ**（直近1時間 ⇄ 起動以降をトグル）
- 4段階状態（正常／注意／警告／危険）に応じたイラスト差し替え
- しきい値跨ぎ時のVOICEPEAK音声再生（ヒステリシス＋クールダウン付き）
- ウィンドウ位置・パネル順序の永続化、ログオン時自動起動

**Phase 1でやらないこと（YAGNI）**
- コア別CPU、常時表示のスパークライン、アイドルアニメ、トレイ常駐UI、設定GUI、キャラスキン切替 → 全てPhase 2送り（※台数追加・並べ替え・最前面切替・クリック式ポップアップグラフはPhase 1に含む）

## 2. 全体アーキテクチャ

```
┌────────────── Ubuntu VPS #i （i = 1..N） ──┐
│  metrics-agent (Go静的バイナリ / PHP代替)   │
│  /proc を直接read（dfやcatをforkしない）    │
│  → stdout に 1行/秒のNDJSONを出力           │
└───────────────┬────────────────────────────┘
                │ SSH (port 49222, 鍵認証, forced-command)
                │ 暗号化トンネル内をNDJSONが流れる
┌───────────────▼──────────────── Windows ───┐
│  VPS Watcher (C# / WPF, .NET LTS)          │
│  ├ ConnectionService ×N (SSH.NET)          │
│  │   ssh exec → stdout を行単位で読む       │
│  ├ MetricsModel / AlertStateMachine        │
│  ├ VoiceService (NAudio, 事前WAV再生)        │
│  └ GadgetWindow                            │
│      └ ItemsControl（パネルを動的生成）      │
│          ・ドラッグで並べ替え                │
│          ・最前面ON/OFFを設定で切替          │
└─────────────────────────────────────────────┘
（VPS #1..#N を同一構成で並列。台数は設定で可変）
```

### 2.1 転送方式の判断：SSH exec ストリーム（NDJSON over stdout）

当初SSE/WebSocket案も検討したが、**Phase 1ではSSH exec方式を主軸**に据える。判断根拠：

| 観点 | SSH exec ストリーム（採用） | localhost常駐デーモン＋トンネル |
|---|---|---|
| 公開ポート | **ゼロ**（listenすらしない） | localhostにlisten（トンネル必須） |
| 既存資産活用 | 49222・鍵認証・fail2banをそのまま流用 | 同左 |
| サーバ側実装 | 約60行のループのみ（Go/PHP） | デーモン＋systemd unit |
| 攻撃面 | 最小（forced-commandで鍵を1コマンドに束縛可） | listenポート分だけ増える |
| 多消費者対応 | 弱い（接続ごとにプロセス） | 強い |

消費者はデスクトップ1台のみ、データは「サーバ→デスクトップ」の純一方向、1Hzで数百バイト。この条件では「接続ごとにプロセス」のデメリットはほぼ無害で、**公開ポートを一切増やさない**メリットが圧倒的。しんさんのセキュリティ志向にも合致する。

> **既知の事実 / 推定の分離**: 「公開ポート不要」「forced-commandで束縛可」は確実。「将来複数の表示先（スマホ等）が欲しくなったら常駐デーモン＋SSE方式へ移行」という拡張は現時点では推定需要なのでPhase 1には入れない。

## 3. サーバ側：metrics-agent（言語非依存）

### 3.1 役割と実装言語
SSH経由で起動され、1秒ごとに1行のNDJSONをstdoutへ出力し続ける常駐ストリーム。systemd登録は**不要**（SSHセッションのライフサイクルに従う）。

**リファレンス実装＝Go静的バイナリ。** 配布容易性・低負荷・アップデート耐性のすべてで最有力（§12参照）。現2台はPHP導入済みのため**PHP版を即席の代替**として併記する。両実装はまったく同じNDJSONスキーマ（§3.4）を出力する契約とし、クライアントは実装言語を区別しない。

| 実装 | 配布 | 低負荷性 | 位置づけ |
|---|---|---|---|
| **Go静的バイナリ** | GitHub Releasesから`curl`で取得・即実行 | ◎ forkなし・常駐RSS数MB・GC軽微 | **本命** |
| PHP CLI | 現環境にphp導入済み | ○ ただしプロセス常駐コストはGoよりやや重い | 現2台の即席代替 |

### 3.2 収集項目と取得元

| 項目 | 取得元 | 計算方法 |
|---|---|---|
| CPU使用率 | `/proc/stat` の `cpu` 行 | busy = total − idle、前回サンプルとの差分比 |
| メモリ | `/proc/meminfo` | `(MemTotal − MemAvailable) / MemTotal`（**MemAvailable基準**。free/buffers/cachedの素朴計算は不正確なので使わない） |
| Swap | `/proc/meminfo` の SwapTotal/SwapFree | 使用率 |
| ストレージ | `statfs(2)` syscall（Goは`syscall.Statfs`） | マウントポイントごと。**`df`をforkしない**のが低負荷の肝 |
| ネットワーク | `/proc/net/dev` | 対象IFのrx/tx_bytesを前回差分→bytes/sec。`lo`は除外 |
| ロードアベレージ | `/proc/loadavg` | 1/5/15分（おまけだが有用） |
| 稼働時間 | `/proc/uptime` | |

> **履歴の保持はクライアント側**：エージェントは各時刻の**瞬時値を1行流すだけのステートレス**を維持する（蓄積しない＝低負荷・単純）。時系列の保持とグラフ化はクライアントの責務（§5.6）。

### 3.3 低負荷のための実装規約（サーバ側）
- **外部コマンドをforkしない**：`df`・`cat`・`ip`等を毎秒起動するとforkコストが積み上がる。すべて`/proc`・`statfs`の直接readで取得する（Go実装の本質的優位点。シェル+awk案を見送る主因でもある）。
- **ファイルディスクリプタを使い回す**：`/proc/stat`等は毎ループ開き直さず、`seek(0)`して再readする（openコスト削減）。
- **バッファ再利用**：行バッファ・パース用スライスを毎ループ再確保せず使い回し、GC圧を最小化。
- **sleepベースの1Hz**：ビジーループ禁止。`sleep`で待つあいだCPUは完全に遊ぶ。
- **ログを吐かない**：正常時は標準エラーにも書かない（ディスクI/Oと容量の無駄を避ける）。異常時のみ最小限。
- 想定フットプリント（推定）：**常駐RSS数MB・アイドルCPU 0%近傍**。1Hzの`/proc`読みは1回あたり数十マイクロ秒オーダー。

### 3.4 実装上の必須ポイント
- **初回サンプルは差分が取れない**：CPUとネットは「2回読んで差分」なので、ループ1周目は前回値の確保だけ行い、レートは `null` で送る（または1周待ってから初回出力）。クライアント側は `null` を「測定中」と表示。
- **ネットワークカウンタのラップ／リセット**：差分が負になったらカウンタ巻き戻し（再起動・IF再設定等）とみなし、その回のレートは `null`。
- **対象インターフェース名**：`eth0` 等は環境依存。設定で明示するか、`/proc/net/route`でデフォルトルートのIFを起動時に1回判定。
- **出力はライン即時フラッシュ**（バッファリングさせない）。

### 3.5 出力スキーマ（NDJSON 1行）

> 下記は**説明用に整形（pretty-print）した表示**であり、実際の出力は**1サンプル＝1行**に連結する（改行・インデントを含めない）。スキーマ契約の正典は `docs/ndjson-schema.md` ＋ `testdata/` の fixture 群。id はダミー（`vps-example-*`）を用い、実IP・実idは書かない。

```json
{
  "v": 1,
  "id": "vps-example-1",
  "ts": 1717300000,
  "cpu_pct": 12.4,
  "mem": { "used_pct": 63.2, "used_mb": 2521, "total_mb": 3989 },
  "swap": { "used_pct": 0.0 },
  "disk": [
    { "mount": "/", "used_pct": 48.1, "used_gb": 24.0, "total_gb": 50.0 }
  ],
  "net": { "iface": "eth0", "rx_bps": 102400, "tx_bps": 51200 },
  "load": [0.12, 0.08, 0.05],
  "uptime_sec": 864000
}
```

- `v`: スキーマバージョン（将来の互換性のため最初から持たせる）
- `id`: サーバ識別子（N台を区別。payloadに必須）
- レート系（cpu_pct, rx_bps, tx_bps）は測定不能時 `null`

### 3.6 起動コマンド（クライアントが実行）
```
# Go版（本命）
/opt/vpswatcher/agent --iface=eth0 --id=vps-tokyo --interval=1
# PHP版（代替）
php /opt/vpswatcher/stream.php --iface=eth0 --id=vps-tokyo --interval=1
```

### 3.7 OS・環境依存マトリクス（重要）

言語よりも**メトリクス取得源と仮想化方式**のほうが環境依存度が高い。新規VPSを足す際は以下を確認する。

| 依存箇所 | 内容 | 対策 |
|---|---|---|
| **procfs前提** | `/proc/*` はLinux固有 | Ubuntu同士なら問題なし。BSD等は対象外 |
| **仮想化方式（最重要）** | KVMは各VMの正値が見えるが、コンテナ型（OpenVZ/LXC）は`/proc`がホスト全体値を返すことがある | KVMを前提（XserverのVPSはKVM）。コンテナ型を足す場合はcgroup（`/sys/fs/cgroup/...`）読みが必要 |
| **NIC名** | `eth0`/`ens3`/`enp1s0`がOS・systemd版で変動 | ハードコードせずデフォルトルートから自動判定＋設定上書き |
| **カーネル版** | `MemAvailable`はkernel 3.14+（2014） | 現行VPSなら確実。極端に古い場合のみフォールバック検討 |
| **マウントポイント** | `/`・`/var/www`等は環境ごとに異なる | 設定で明示指定 |
| CPU%計算 | USER_HZ非依存（差分比のため） | **環境非依存**。対策不要 |

## 4. サーバ側セキュリティ設計

しんさんの既存ハードニング（SSH 49222・`PermitRootLogin no`・fail2ban・鍵認証）を前提に、さらに以下を重ねる。

1. **専用の低権限ユーザ** `metrics` を作成。`/proc` は基本world-readableなので **sudo不要**・root不要で全項目取得可能。
2. **authorized_keys の forced-command**：監視用公開鍵を、エージェント起動のみに束縛する。万一鍵が漏れてもシェルは取れない。
   ```
   command="/opt/vpswatcher/agent --iface=eth0 --id=vps-tokyo",no-pty,no-port-forwarding,no-X11-forwarding ssh-ed25519 AAAA... metrics@watcher
   ```
3. **公開ポートは増やさない**：エージェントはlistenしない。SSH 49222のみ。
4. **クライアント側で known_hosts ピン留め**：MITM防止のためホスト鍵を固定検証（後述）。

> この forced-command 方式により「監視鍵＝メトリクス閲覧専用」が成立する。鍵分離はしんさんのSetGID/umask設計と同じ「権限の最小化」思想の延長。

### 5.2 ウィンドウ仕様（ガジェット）
```xml
WindowStyle="None"
AllowsTransparency="True"
Background="Transparent"
Topmost="{Binding AlwaysOnTop}"   <!-- 設定でON/OFF。後述 -->
ShowInTaskbar="False"
ResizeMode="NoResize"
```
- ドラッグ移動：`MouseLeftButtonDown` → `DragMove()`
- 位置・パネル順序の永続化：`%APPDATA%\VpsWatcher\state.json` に保存・復元
- イラストPNGのアルファは `AllowsTransparency=True` でそのまま抜ける

#### 5.2.1 最前面常駐の切替（設定）
- `Topmost` を設定値 `alwaysOnTop`（真偽）にバインドし、実行中のトグルでも即反映する。
- **ON**：他ウィンドウの上に常駐（従来の挙動）。
- **OFF**：通常ウィンドウとして背面に回れる（作業中は隠れ、必要時にクリックで前面化）。
- 設定はキャラクター・モニター情報パネルを含む**ウィンドウ全体**に効く（キャラと数値は同一ウィンドウ内に配置するため、片方だけ最前面という状態は作らない）。
- 任意の補助：OFF時でも「警告/危険へ昇格した瞬間だけ一時的に前面化」するオプションをPhase 2候補として残す（Phase 1は単純なON/OFFのみ）。

#### 5.2.2 パネルの動的生成と並べ替え
- サーバパネルは固定数ではなく **`ItemsControl`（`ObservableCollection<ServerViewModel>`）** で動的生成。設定の追加だけでN台に増える。
- レイアウトは縦積み（`StackPanel`）を基本とし、各パネルが1台分のキャラ＋数値ブロックを表示。
- **ドラッグ並べ替え**：パネルを掴んでドラッグし順序入替。実装方針は §5.5 に記載。
- 並べ替え結果（`order`）は即 `state.json` に永続化し、次回起動時に復元。

### 5.3 コンポーネント構成
- **ConnectionService（サーバ毎に1インスタンス、計N個）**
  - SSH.NETで接続 → `RunCommand` のストリームをstdout行単位で読む
  - 1行受信ごとにJSONデシリアライズ → `MetricsReceived` イベント発火
  - **再接続ロジック**：EOF／例外検知で待機後リトライ（指数バックオフ、上限あり）
  - **死活検知**：一定秒（例10秒）受信が途絶えたら `Disconnected` 状態へ
- **ServerViewModel（サーバ毎）**：最新値・状態・表示順を保持しUIへバインド。`ObservableCollection<ServerViewModel>` で全台を管理。
- **AlertStateMachine（サーバ毎にインスタンス）**：しきい値判定（後述）
- **VoiceService（アプリ単一）**：各サーバの状態遷移イベントを受けてWAV再生（多重発音制御は全台横断）
- **GadgetWindow**：`ItemsControl` でServerViewModelを動的描画。`AlwaysOnTop`設定を保持。

> 受信スレッド（SSH読み取り）からUIへは`Dispatcher`経由でマーシャリング。低負荷のため**UI更新は値が変わったプロパティのみ**を通知し（`INotifyPropertyChanged`）、毎秒の全画面再描画は行わない（詳細は§13）。N台でも各パネルは独立更新なので、台数増加に対し描画コストは線形かつ部分的。

### 5.4 接続断のハンドリング
- 受信途絶 → `Disconnected` 状態。イラストを専用差分にし、任意で「接続が切れました」ボイス。
- 復帰 → 「復帰しました」ボイス。**切断自体を警告トリガーにできる**のがこの構成の利点。

**検出は client 内部の wall-clock ウォッチドッグで行う（スキーマ外）。**
- SSH の **EOF／例外で即時 `Disconnected`**。加えて、最終受信からの**無音が閾値秒（例10秒）継続**しても `Disconnected`。判定は純粋に**ローカル時計（wall-clock）**で行う。
- **`ts` では生死判定しない**：断中は新しい `ts` が来ず、サーバ時計のずれも混入し得るため、`ts`（サーバ打刻・履歴軸の正、§5.6.2）を死活判定に使わない。`ts` は履歴の時間軸、接続断は別系統の検出、と責務を分ける。
- **NDJSON スキーマには載せない**：エージェントはステートレスで、落ちたら何も送れない（自己申告不可）。接続断は完全にクライアント状態であり、スキーマに断のフィールドを足さない（`docs/ndjson-schema.md` §8）。
- **履歴との接点**：断中はサンプルが来ないため該当スロットは穴（`null`）として残り（§5.6.2）、復帰後のグラフは欠測区間が途切れ／点線で表現される。検出ロジックは別だが**見え方は一貫**する。

### 5.5 パネル並べ替えの実装方針（gong-wpf-dragdrop採用・確定）
- **採用ライブラリ**：`gong-wpf-dragdrop`（GongSolutions.WPF.DragDrop、BSD 3-Clause、NuGet）。「依存より安定性」の方針に基づき、自前実装ではなく枯れたライブラリを採用する。WPF素のD&D（OLE基盤）はクリック/ドラッグ判別・挿入アドナー・キャンセル処理などエッジケースが多く、自前は堅牢化が工数依存になるため。
- **依存リスクの中和**：self-contained単一ファイル発行により、当ライブラリのDLLはビルド時点のバージョンで成果物に**凍結**される。OS/.NETが裏で更新されても同梱依存は不変。更新は将来の.NETメジャー版移行時の**意図的な保守操作**のみ。
- **MVVM適合**：添付プロパティでViewModelのドロップハンドラにバインドし、コードビハインドを書かずに済む。`ObservableCollection<ServerViewModel>`に対する同一コントロール内並べ替えに標準対応。
- **判定/視覚**：挿入位置のアドナー表示・ドロップ位置判定はライブラリ既定に委ねる。
- **低負荷との両立**：並べ替えはユーザー操作時のみの一過性イベント。常駐コストは増えない（§13の常駐アニメ禁止原則に抵触しない）。
- **永続化**：ドロップ確定時に各`ServerViewModel.Order`を再採番し`state.json`へ保存。
- **DragMoveとの競合回避**：パネル並べ替えと、透過ウィンドウ自体の`DragMove`（ウィンドウ移動）が衝突しないよう、**並べ替えは専用のドラッグハンドル（パネル左端のグリップ領域）からのみ開始**する。gong側で`DragHandler`の開始対象をそのグリップに限定する。

### 5.6 メトリクス履歴とポップアップグラフ

各メトリクス項目をクリックすると、その項目の時系列グラフをポップアップ表示する。グラフをクリックするたびに「直近1時間」⇄「アプリ起動以降の全履歴」をトグルする。

#### 5.6.1 履歴の保持場所と方針
- **履歴はクライアント（WPF）側で保持する。** サーバ側エージェントは引き続き**瞬時値を流すだけのステートレス**を維持し（§3の低負荷・単純性を崩さない）、蓄積はクライアントの責務とする。
- サーバ／メトリクスごとに独立した時系列を`ServerViewModel`配下に持つ（CPU%・メモリ%・ストレージ%・ネットrx/tx 等）。
- **起動以降の履歴**＝アプリ起動時刻が起点。ログオン自動起動のため、PC再起動で自然にリセットされる（実運用上の上限が入る）。

#### 5.6.2 メモリ設計（低負荷原則との両立がここの肝）
1Hzで貯め続けると「最小メモリ」原則と衝突するため、**2層（リング＋ロールアップ）**で保持する。

- **`ts` 基準のスロット割り付け（時間軸の正＝`ts`）**：時刻は到着順ではなく、サーバが各サンプルに打刻した **`ts`（Unix epoch 秒・UTC）を正**とする（`docs/ndjson-schema.md` §7）。ネット遅延・欠落・再接続で到着順がズレるため、**到着順では並べない**。各サンプルは `slot = ts − t0` で固定長リングのスロットへ配置する（`t0` は窓の基準時刻）。**メモリ設計は不変**：スロットには数値のみを格納し、タイムスタンプを全サンプル分持つことはしない。到着順インデックスから **割り付けのキーを `ts` に変えるだけ**で、固定長・低メモリの利点はそのまま。
  - **欠落＝穴（欠測 `null`）**：その `ts` のサンプルが来なければスロットは空のまま。データのある区間と明確に区別し、グラフは欠測区間で線を途切れさせる（または点線）。**左詰めにしない**。
  - **遅延・バースト＝再配置**：まとめて届いても各サンプルを自分の `ts` のスロットへ置く。**重複（同一 `ts`）は後着で上書き（last-wins・確定）**：後に届いたサンプルでスロットを上書きする（再送・再接続時の新しい値が正。`docs/ndjson-schema.md` §7）。
  - **時計ずれ**：サーバの `ts` をそのまま採用（サーバ時刻基準で表示）。各パネルは独立なので**複数サーバ間の時刻整合は不要**。なお接続断の死活判定は `ts` ではなく client の wall-clock で行う（§5.4）。
- **直近1時間：固定長リングバッファ**（3600スロット = 1スロット1秒）。`float`なら 1系列あたり 3600×4B ≒ **14KB**。「直近1時間」表示はこれを直接描画（欠測スロットは穴として描く）。
- **起動以降：1分ロールアップ**。リングからこぼれたサンプルは1分バケットへ集約（min/avg/max の3値）。1日でも 1440バケット×12B ≒ **17KB/系列**。「全履歴」表示はこのロールアップ＋直近リングを連結して描画。
- 概算：5系列 × N台 でも、数台規模なら合計**数百KB〜数MB**に収まる。万一極端な長時間運用でロールアップが嵩む場合に備え、総量上限（超過時は古いバケットをさらに粗く集約）を設定値で持てるようにする。

> **設計判断**：これはRRDtool等の監視系で定番の階層集約と同じ発想。「全履歴」でも古い区間は1分粒度になるが、長期の傾向把握には十分で、メモリは有界に保てる。

#### 5.6.3 UIインタラクション
- **項目クリック**：CPU/メモリ/ストレージ/ネットの各行をクリック可能要素にする。クリックで`Popup`を開き、その項目の**直近1時間グラフ**を表示。
- **グラフクリックでトグル**：グラフ領域のクリックで `LastHour ⇄ SinceLaunch` を切替。もう一度で戻る。Popup内にViewState（enum）を持ち、クリックで反転して再描画。
- **閉じる**：`Popup` を `StaysOpen=false` にし、外側クリックで自動的に閉じる（軽量・実装単純）。
- 表示中はどのモードかを小さく明示（「直近1時間 / 起動以降」）。

#### 5.6.4 グラフ描画方式
- **`Popup`コントロール**を使用（独立HWNDのため最前面ウィンドウより上に出る。位置はクリック要素に相対配置）。**ポップアップが開いている間だけ描画**されるので、閉じている間のコストはゼロ。
- **描画は自前ベクター（`StreamGeometry` ＋ `Path`）を主軸**にする。単純な折れ線時系列はこれで十分軽量・依存ゼロで、self-contained単一ファイルとも相性が良い。
- **ビューポート間引き（重要）**：描画点数を**ポップアップ横幅のピクセル数以下に間引く**（例：横300pxなら最大～300点へmin/maxデシメーション）。「全履歴」で点数が多くても、画面幅以上は描かない。これで長期履歴でも描画は一定コスト。
- **更新頻度**：開いている間のライブ更新は最大1Hz（直近1時間モードでは末尾1点を追加するだけ）。閉じれば更新も停止。
- **チャートライブラリを使う場合（任意）**：軸・目盛り・ツールチップを楽に得たいなら採用可。その際は**OxyPlot（純マネージドWPF・ネイティブ依存なし）を推奨**。ScottPlot 5 / LiveCharts2 はSkiaSharp（ネイティブ）依存で、self-contained単一ファイル発行時にネイティブアセット同梱の手当てが要るため、当プロジェクトの配布方針とは相性がやや劣る（※各ライブラリの最新の単一ファイル対応状況は採用時に確認推奨）。
- **しきい値の水平線（オーバーレイ）**：折れ線に重ねて、注意／警告／危険の3本の水平基準線を描く。履歴上で「いつ危険域へ入ったか」が一目で分かる。
  - **値の出所**：線の高さは§6.2／§9.1の**設定値**（`servers.json`のサーバ別`thresholds`）を参照する。**サーバ単位で変更可能**なので、線の位置もサーバごとに異なり得る。設定変更時はポップアップ再描画で即反映。
  - **対象メトリクス**：CPU%・メモリ%・ストレージ%・Swap%など百分率系に表示。**ネットワークは既定でしきい値なし（§6.2）**のため線は描かない（設定で上限を与えた場合のみ描く）。
  - **目盛り（縦軸）**：百分率系は **0〜100%固定スケール**にすると線の意味が安定する（自動スケールにすると基準線の見かけ位置が変動して読みにくいため）。ネット等の非百分率系のみ自動スケール。
  - **配色（状態色と統一）**：注意＝黄、警告＝橙、危険＝赤の破線。本体の折れ線とは線種・濃度で区別し、低負荷のため線は静的（点滅等はしない）。
  - **実装**：基準線も`StreamGeometry`の水平線として本体折れ線と同一Canvas上に重畳。3本＋折れ線で描画要素はごく少数。OxyPlot採用時は`LineAnnotation`で表現。

#### 5.6.5 低負荷チェックリスト（履歴機能）
- 蓄積は1Hzで配列末尾へ1要素追加＋リング更新のみ（O(1)）。グラフ描画は**ポップアップ表示中のみ**。
- 履歴更新でUI通知を出さない（数値表示の更新とは別経路。グラフは開いている時だけ購読）。
- 間引きにより描画点数を画面幅で頭打ちにする。

## 6. アラート状態機械（設計の最重要部）

### 6.1 状態
`Normal → Caution → Warning → Critical`（＋ `Disconnected`）
ガジェット全体の状態 = **全メトリクスの最悪値**（worst-of）。

### 6.2 しきい値（初期値・設定で変更可）

| メトリクス | 注意 | 警告 | 危険 |
|---|---|---|---|
| CPU% | 70 | 85 | 95 |
| メモリ% | 75 | 88 | 95 |
| ストレージ% | 80 | 90 | 95 |
| Swap% | 25 | 50 | 80 |
| ネットワーク | 原則アラートなし（表示のみ。必要なら設定で上限指定） | | |

### 6.3 フラつき対策（ここが肝）
境界値でCPUが振動して喋り続ける事故を防ぐため、3重の防御：

1. **ヒステリシス（enter/exit分離）**：例）警告へ入るのは85%超、警告から出るのは80%未満。入口と出口を別しきい値にする。
2. **デバウンス（連続サンプル要求）**：例）3秒連続で超過して初めて昇格。1秒スパイクは無視。
3. **クールダウン**：同一アラートは一定時間（例5分）再生しない。

### 6.4 音声トリガー条件
- **状態の昇格時のみ**発火（Normal→Caution 等）。毎フレーム発火しない。
- 降格（回復）時は任意で軽い回復ボイス1回。
- サーバ別に状態とクールダウンを独立管理（N台のどれが昇格しても個別制御）。台数が増えても警告が混線しないよう、ボイスは発話キューで直列化し、同時多発時は重要度（Critical優先）で順序付け。

### 9.1 サーバ定義 `%APPDATA%\VpsWatcher\servers.json`
配列なので、要素を足すだけでN台に拡張できる（台数のハードコードなし）。
```json
[
  {
    "id": "vps-tokyo",
    "label": "東京 (ViPass)",
    "host": "203.0.113.10",
    "port": 49222,
    "user": "metrics",
    "keyPath": "%USERPROFILE%\\.ssh\\watcher_ed25519",
    "knownHostKey": "AAAA...",        // ホスト鍵ピン留め
    "iface": "eth0",
    "mounts": ["/", "/var/www"],
    "thresholds": { "cpu": [70,85,95], "mem": [75,88,95], "disk": [80,90,95], "swap": [25,50,80] }
  },
  { "id": "vps-osaka", "label": "大阪 (EC)", "host": "...", "port": 49222, "...": "..." }

```
- `label`：パネル見出し用の表示名（任意。未指定なら`id`）。
- しきい値・IF名・マウントポイントを明示設定（しんさんの明示config志向に合わせる）。

```json
{
  "alwaysOnTop": true,            // 最前面常駐のON/OFF（§5.2.1）
  "windowPosition": { "x": 1600, "y": 80 },

```
- `order`：`servers.json` の `id` を表示順に並べた配列。ドラッグ並べ替えのたびに更新。`servers.json`に存在しないidは無視、未掲載のidは末尾に自動追加（台数増減に追従）。

## 10. 配布（GitHub Releasesから直接ダウンロード）

「落として即実行」を最優先に、両コンポーネントをGitHub Releasesで配る。

### 10.1 サーバ・エージェント（Go静的バイナリ）
- `CGO_ENABLED=0` で完全静的ビルド → glibc非依存でどのUbuntuでも動く。
- `GOOS=linux GOARCH=amd64` と `arm64` の2種をRelease添付。
- VPS側はワンライナーで導入：
  ```
  curl -fsSL https://github.com/<user>/<repo>/releases/latest/download/agent-linux-amd64 \
    -o /opt/vpswatcher/agent && chmod +x /opt/vpswatcher/agent
  ```
- arch自動判定する `install.sh` を1本添えるとさらに親切。
- **SHA256チェックサムを同梱**し、導入後に検証（改ざん検知。しんさんのセキュリティ志向に合わせる）。

### 10.2 Windowsクライアント（self-contained単一exe）
- `dotnet publish -r win-x64 --self-contained -p:PublishSingleFile=true` で単一exe → Release添付 → ダウンロードして実行で完結。
- **正直な注意点2つ**：
  - **サイズ大**：self-containedは.NETランタイム同梱で100MB超になりやすい。WPFは**トリミング・NativeAOTが実質非対応**なので縮められない。“ゼロインストール”を取るならこのサイズは受け入れる（framework-dependentなら数MBだが「.NETを入れて」が発生する）。
  - **SmartScreen警告**：署名なしexeは初回に保護警告が出る。自分用／少人数なら無視で可。広く配るならコードサイニング証明書（有料）が必要。

### 10.3 CI（GitHub Actions・タグ連動）
```yaml
on: { push: { tags: ['v*'] } }
jobs:
  agent:    # ubuntu-latest: go build を GOARCH=amd64,arm64 でマトリクス → 添付
  client:   # windows-latest: dotnet publish (win-x64, self-contained, single-file) → 添付
  # 両jobの成果物を softprops/action-gh-release で同一Releaseへ。SHA256も生成・添付。
```

### 10.4 自動起動
- **タスクスケジューラ「ログオン時」**（遅延起動可・最も堅牢）。Startupフォルダ／Runキーでも可。

## 11. リソース見積り（推定）

| 対象 | アイドル時CPU | メモリ | 備考 |
|---|---|---|---|
| サーバ・エージェント（Go） | ほぼ0% | 常駐RSS数MB | 1Hzの`/proc`読みは数十μs/回・forkなし |
| サーバ・エージェント（PHP） | ほぼ0% | 十数〜数十MB | PHPプロセス常駐分でGoよりやや重い |
| sshd（接続1本あたり） | 微小 | 数MB | 数百B/秒の暗号化は誤差 |
| Windowsクライアント | ほぼ0%（後述の制御込み） | 数十MB＋ランタイム | 1Hzパース・部分再描画のみ |

> **N台時のスケール（推定）**：クライアント側はSSH接続・パース・パネル描画が台数に対し概ね線形。各パネルは独立・部分更新なので、数台〜十数台規模なら常駐負荷は依然として小さい。多数台（数十）になればSSH接続本数とメモリが効いてくるため、その規模が現実になったら「常駐デーモン＋SSE集約」への移行（§12）を検討する。サーバ側は各VPSで独立しているため台数の影響を受けない。

## 13. 低負荷設計の集約（基本原則①の具体策）

サーバ・ローカル両方で「アイドル時CPUほぼ0%・メモリ最小」を満たすための具体策をここに集約する。

### 13.1 サーバ側（§3.3の再掲＋補足）
- 外部コマンドを**forkしない**（`/proc`・`statfs`直接read）。← Go採用の主因
- FDを使い回し（`seek(0)`再read）、バッファ再確保を避けてGC圧を抑制。
- `sleep`ベースの1Hz（ビジーループ禁止）。
- 正常時はログ出力なし（ディスクI/O節約）。
- 送信ペイロードは数百B/秒に収め、SSH暗号化負荷を誤差に保つ。

### 13.2 ローカルPC側（WPF）
- **イベント駆動・1Hz更新のみ**：受信時だけ処理。ポーリングやタイマー連打をしない。
- **部分再描画**：`INotifyPropertyChanged`で変化したプロパティだけ通知。毎秒の全画面Invalidateをしない。バーは数値バインドで更新し、ジオメトリ再生成を避ける。
- **イラスト差し替えは状態遷移時のみ**：毎フレームの`Image.Source`差し替えをしない。各PNGは起動時に1回だけデコードして`BitmapImage`をキャッシュ（`BitmapCacheOption.OnLoad`＋`Freeze()`）。
- **音声リソースは1回ロード**：WAVを起動時に読み込み、再生のたびに再デコードしない。多重再生防止で同時発音数を抑制。
- **アニメーションを常駐させない**：Phase 1は静止画。`Storyboard`常時再生やDispatcherTimerでの常時アニメは避ける（CPUとGPUの常時消費要因）。
- **GCとアロケーション**：1Hzのため通常は問題にならないが、受信パスでの不要なstring結合・LINQ多用を避け、Gen0 GC頻度を下げる。
- **透過ウィンドウの描画コスト**：`AllowsTransparency=True`のレイヤードウィンドウは中身が静止していれば再描画は発生しない。常時動くエフェクトを置かないことで実質ゼロ負荷を維持。
- **プロセス優先度**：必要なら`BelowNormal`に下げ、他作業を最優先にする（任意）。
- **最前面OFF時**：`Topmost=false`でも描画コストは変わらない（静止していれば再描画なし）。前面化判定の常時ポーリング等は行わず、通常のウィンドウZオーダーに委ねる。
- **履歴（§5.6）**：タイムスタンプ非保持＋リング（直近1h）＋1分ロールアップで有界化。蓄積はO(1)、グラフ描画はポップアップ表示中のみ＋画面幅へ間引き。常駐メモリは数百KB〜数MBに収める。

> 結果として、しんさんの開発作業中もガジェットの存在がCPU・メモリ統計にほぼ現れないことを設計目標とする。

## 15. 開発体制・実装フロー

### 15.1 実装手段と役割分担
- 実装は **Claude Code** が行い、設計者（しん）と Claude（本チャット）が**監査・指示役**を務める。
- **ランタイムで分業**する（「テストできる場所で書く」原則）：
  - **VPS側 Claude Code（dev VPS）**：Goエージェント。`/proc`・`statfs`の実機検証ができるLinux環境が必須なため。
  - **Windows側 Claude Code**：WPFクライアント。WPFはWindows専用でビルド・実行・透過ウィンドウ確認がWindowsでしかできないため。
- 監査担当の割当（しんの既存サブエージェントを流用）：
  - `consistency-checker` → NDJSONスキーマ契約のドリフト検出。
  - `critical-security-reviewer` → SSH周り（forced-command・known_hostsピン留め・秘密情報の非コミット）。

### 15.2 リポジトリ構成（モノレポ）
結合した2コンポーネント＋共有契約のため**単一リポジトリ**にする。
```
vps-monitor-mei/
├─ agent/                 # Goエージェント（VPS側Claude Codeが主担当）
├─ client/                # WPFクライアント（Windows側Claude Codeが主担当）
├─ docs/
│   ├─ design.md          # 本設計書
│   └─ ndjson-schema.md   # スキーマ契約（単一の真実）
├─ testdata/              # ゴールデンサンプル（両側のテストが参照する3系統）
│   ├─ sample.ndjson            # 正常
│   ├─ sample_measuring.ndjson  # 測定中（レート系 null）
│   └─ sample_multidisk.ndjson  # 複数 disk（2要素）
├─ .github/workflows/     # CI（タグ連動ビルド＆Release）
├─ servers.example.json   # 設定テンプレ（実値は含めない）
├─ .gitignore
├─ LICENSE
├─ CHANGELOG.md
└─ README.md
```

### 15.3 契約のドリフト防止（分業の最大リスク対策）
- **スキーマを単一の真実に**：`docs/ndjson-schema.md` ＋ `testdata/` の3 fixture（`sample.ndjson`＝正常 / `sample_measuring.ndjson`＝測定中(null) / `sample_multidisk.ndjson`＝複数disk）。詳細は `ndjson-schema.md` §6。
- 両側で**同じfixtureに対するテスト**を持つ：Goは「3 fixture に準拠した出力を吐く」テスト、WPFは「3 fixture（正常・null・複数disk）を正しくパースする」テスト。
- **CIが番人**：単一リポジトリから両方をビルド。スキーマがズレればクロステストがCIで落ちる。
- **進め方**：先にVPS側でエージェント＋スキーマを凍結しゴールデンサンプルを産出 → Windows側はそれに対しオフライン開発 → 最後にdev VPSへ実SSH接続で結合テスト（本番と同じ経路）。契約凍結後は両セッションが概ね独立に走れる。

### 15.4 Git / GitHub 運用（公開配布アプリ向け）
個人開発の私的リポジトリと違い、**公開・配布物**であるため「mainが良好」だけでは不十分で、**利用者が落とすのはRelease（タグ由来）**である点が新しい。

- **ブランチモデルは GitHub Flow（軽量）** を採用。重厚なGitFlow（develop/release/hotfix）は単独開発には過剰。
  - `main` = **常にビルド可能な統合ブランチ**。
  - 機能・修正は**短命なフィーチャーブランチ**を切り、PR経由で`main`へ。PRは監査の自然な単位（Claudeがdiffレビュー）。
- **公開物はタグから作る**：`main`の良コミットに **SemVer タグ**（`vMAJOR.MINOR.PATCH`）を打つ → CI（§10.3）が発火 → Go各アーキ＋WPF self-contained をビルドし、**GitHub Release**に成果物＋SHA256を添付。**利用者はReleaseからダウンロード**（cloneではない）。
  - 公開前検証には**プレリリースタグ**（`v0.1.0-rc1`）を活用。
- **「main＝公開用」の正確な理解**：main自体が配布物ではなく、*mainからタグで切り出したRelease*が配布物。この区別が私的開発との最大の差。

### 15.5 公開リポジトリで初めて要るもの（チェックリスト）
- **`.gitignore` ＋ 秘密情報の非コミット（最重要）**：公開リポジトリのため、**実サーバのIP・SSH鍵・known_host値・`servers.json`の実値を絶対にコミットしない**。リポジトリには `servers.example.json`（ダミー値）のみ置く。ビルド成果物・`bin/`・`obj/`も除外。
  - ※私的開発では設定コミットは無害だったが、公開では即漏洩。発想の切替が必要。
- **`LICENSE`**：配布するのでライセンス明示が必要。アプリ本体は例えばMIT。依存の `gong-wpf-dragdrop`（BSD 3-Clause）・`OxyPlot`（MIT）等とも両立。
- **`README.md`**：導入手順（エージェントの`curl`ワンライナー §10.1、SHA256検証、Windowsクライアントのダウンロードと**SmartScreen警告の説明** §10.2）、設定方法、スクリーンショット。
- **`CHANGELOG.md`**：利用者がバージョン差分を把握できるよう、タグごとに変更を記載。
- **VOICEPEAK音声・キャライラストのライセンス確認**：同梱して公開配布する場合、各素材の再配布可否を確認（VOICEPEAKはクレジット不要だが配布形態の規約は要確認＝推定扱い）。

## 付録A：CPU使用率の差分計算（擬似コード）
```
prev = read_cpu()            // user,nice,system,idle,iowait,irq,softirq,steal
sleep(interval)
cur  = read_cpu()
idle_d  = (cur.idle + cur.iowait) - (prev.idle + prev.iowait)
total_d = sum(cur) - sum(prev)
cpu_pct = total_d > 0 ? (1 - idle_d/total_d) * 100 : null
```
